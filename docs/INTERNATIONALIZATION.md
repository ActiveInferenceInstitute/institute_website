# Internationalization & Locales

## Overview

The institute_website is a fully multilingual static site built at compile time. English is the canonical source; every other language is pre-rendered at build time into its own URL subtree (`/<code>/…`), with a language switcher on every page. This design ensures:

- **Proper SEO**: Full `hreflang` alternates, `<html lang>` attributes per locale, correct canonical URLs
- **Instant, offline-ready**: No runtime translation or API calls (CSP forbids `connect-src` entirely)
- **Reviewable translations**: All translation catalogs are committed JSON, reviewable in pull requests
- **Deterministic builds**: The build process is pure—no model runs during `npm run build`

The only place a translation model runs is the **offline** script `scripts/i18n_translate.mjs`, which you invoke deliberately.

## Locale List & Registry

Locales are registered in **`src/i18n/locales.json`** with the following structure:

```json
{
  "defaultLocale": "en",
  "locales": [
    { "code": "en", "name": "English", "nativeName": "English", "dir": "ltr", "machine": false },
    { "code": "es", "name": "Spanish", "nativeName": "Español", "dir": "ltr", "machine": true },
    { "code": "fr", "name": "French", "nativeName": "Français", "dir": "ltr", "machine": true },
    { "code": "de", "name": "German", "nativeName": "Deutsch", "dir": "ltr", "machine": true },
    { "code": "pt", "name": "Portuguese", "nativeName": "Português", "dir": "ltr", "machine": true },
    { "code": "it", "name": "Italian", "nativeName": "Italiano", "dir": "ltr", "machine": true },
    { "code": "ru", "name": "Russian", "nativeName": "Русский", "dir": "ltr", "machine": true },
    { "code": "zh", "name": "Chinese", "nativeName": "中文", "dir": "ltr", "machine": true },
    { "code": "ja", "name": "Japanese", "nativeName": "日本語", "dir": "ltr", "machine": true },
    { "code": "ko", "name": "Korean", "nativeName": "한국어", "dir": "ltr", "machine": true },
    { "code": "hi", "name": "Hindi", "nativeName": "हिन्दी", "dir": "ltr", "machine": true },
    { "code": "ar", "name": "Arabic", "nativeName": "العربية", "dir": "rtl", "machine": true }
  ]
}
```

**12 locales total**: English (default) + Spanish, French, German, Portuguese, Italian, Russian, Chinese, Japanese, Korean, Hindi, Arabic.

| Field | Meaning |
| --- | --- |
| `code` | BCP-47 language code; becomes URL prefix (`/es/`, `/fr/`, etc.) and `<html lang>` attribute |
| `name` | English name; also the target language name the translator prompts toward |
| `nativeName` | Shown in the language switcher dropdown |
| `dir` | `ltr` (left-to-right) or `rtl` (right-to-left, e.g., Arabic); sets `<html dir>` |
| `machine` | `true` shows a machine-translation provenance notice linking to the English original; set `false` after human review |

## Locale String Storage

All translatable strings are stored as **committed JSON** catalogs in `src/content/i18n/`:

```
src/content/i18n/
├── _strings.json       (extracted work list: all unique English source strings)
├── ar.json             (Arabic translations)
├── de.json             (German translations)
├── es.json             (Spanish translations)
├── fr.json             (French translations)
├── hi.json             (Hindi translations)
├── it.json             (Italian translations)
├── ja.json             (Japanese translations)
├── ko.json             (Korean translations)
├── pt.json             (Portuguese translations)
├── ru.json             (Russian translations)
└── zh.json             (Chinese translations)
```

### Catalog Format

Each locale catalog (e.g., `es.json`) is a flat JSON object mapping English source strings to translations:

```json
{
  "10 compact official destinations": "10 destinos oficiales compactos",
  "About": "Acerca de",
  "A 2023 course — 'Active Inference for the Social Sciences' — was produced…": "Un curso de 2023 - «Inferencia Activa para las Ciencias Sociales» - fue producido…",
  …
}
```

Keys are the **exact English source strings** as they appear in the code (wrapped in `tr()`). Values are translations. If a key is missing from a catalog, the build gracefully falls back to English via the `tr()` function.

### Extracted Strings File

`src/content/i18n/_strings.json` is a flat array of every unique English source string that appears in the codebase:

```json
[
  "10 compact official destinations",
  "17 research and paper rows",
  "About",
  "A 2023 course — 'Active Inference for the Social Sciences'…",
  …
]
```

This file is generated by `npm run i18n:extract` and serves as the work list for the translator.

## How Locale Strings Are Used at Build Time

### `tr()` Function

Located in **`src/i18n/index.mjs`**, the `tr()` function is the single access point for translations:

```js
import { tr } from "../i18n/index.mjs";

export function myComponent() {
  return `<h2>${escapeHtml(tr("Open Source Map"))}</h2>`;
}
```

**Behavior**:
- If the active locale is English (default), returns the source string unchanged
- If the active locale is non-default, looks up the source string in the active locale's catalog (e.g., `es.json`)
- If the key is found and non-empty, returns the translation
- If the key is missing or empty, falls back to the English source (graceful degradation)
- In extraction mode (`I18N_EXTRACT=1`), records every source string for the translate pipeline

**Rules for wrapping**:
- Wrap **each visible string once**, at the point the final English text exists
- Do **not** translate DOM IDs, anchors, URLs, slugs, or class names
- Helpers like `sectionHeading`, `cardGrid`, `breadcrumb`, `pageGuide` already call `tr()` internally—pass them raw English
- For interpolated values: `tr("View all {n} public repositories").replace("{n}", count)` so the whole sentence translates

### Active Locale Context

The build switches the active locale once per pass through all pages:

```js
// In src/build.mjs
for (const locale of LOCALES) {
  setActiveLocale(locale.code);
  // Render every routed page once; tr() now returns translations for this locale
  for (const [slug, render] of slugRenderers) {
    writeFile(outputPathForSlug(slug), render());
  }
}
```

This means **every page is rendered once per locale**, and all `tr()` calls inside that render return the appropriate translation.

## How the Translate Script Works

**`scripts/i18n_translate.mjs`** is the offline translator. It runs **outside the build** and produces the JSON catalogs that the build then reads.

### Workflow

1. **Extract**: Run `npm run i18n:extract` to collect all `tr()` source strings into `_strings.json`
2. **Translate**: Run `npm run i18n:translate --locale <code>` to translate missing keys
3. **Build**: Run `npm run build` to render every locale using the committed catalogs
4. **Check**: Run `npm run check` to verify the build

### Translation Providers

The translator supports two backends, configurable via `I18N_PROVIDER`:

#### Ollama (Default)

Local, free, no API key required. Requires `ollama serve` running with a model pulled:

```bash
# Pull models (CJK-optimized + generalist)
ollama pull qwen2.5:3b    # for Chinese, Japanese, Korean
ollama pull gemma3:4b     # for all other languages

# Translate one locale
npm run i18n:translate -- --locale es

# Translate all non-default locales
npm run i18n:translate -- --all

# Override model per-run
npm run i18n:translate -- --locale zh --model qwen2.5:3b

# Re-translate from scratch
npm run i18n:translate -- --locale de --force

# Test with a limit
npm run i18n:translate -- --locale fr --limit 50
```

Per-locale model defaults (from `DEFAULT_MODEL_BY_LOCALE` in the script):
- **CJK** (`zh`, `ja`, `ko`): `qwen2.5:3b` (stronger on non-Latin scripts)
- **All others**: `gemma3:4b` (solid multilingual generalist)

For higher quality, pull a larger model like `aya-expanse:8b` and pass `--model`:

```bash
ollama pull aya-expanse:8b
npm run i18n:translate -- --all --force --model aya-expanse:8b
```

#### Hosted OpenAI-Compatible API

For higher quality or when no local Ollama is available:

```bash
# OpenAI
I18N_PROVIDER=openai I18N_API_KEY=sk-... I18N_API_MODEL=gpt-4o-mini \
  npm run i18n:translate -- --all

# OpenRouter (e.g., Claude, Llama, Mistral)
I18N_PROVIDER=openai \
  I18N_API_BASE=https://openrouter.ai/api/v1 \
  I18N_API_KEY=$OPENROUTER_KEY \
  I18N_API_MODEL=anthropic/claude-3.5-sonnet \
  npm run i18n:translate -- --all
```

Configuration environment variables:
- `I18N_PROVIDER`: `ollama` (default) or `openai`
- `I18N_API_BASE`: API endpoint (default `https://api.openai.com/v1`)
- `I18N_API_KEY`: Required for hosted APIs
- `I18N_API_MODEL`: Model identifier (default `gpt-4o-mini`)

### Translation Process

Both providers use the **same prompt and cleanup strategy**:

1. **Delimiter-framed prompt**: Each source string is wrapped in guillemets (`«…»`) with an engine-style instruction to translate **only** the wrapped text. This prevents verbose models from answering or expanding short labels.

2. **Output sanitization** (`cleanTranslation`):
   - Cuts at the first line break (every source string is single-line)
   - Strips echoed wrapping (guillemets, CJK brackets, quotes, bold/italic markers)
   - Removes leaked prefixes like `English:`, `<Language>:`, `Translation:`
   - Preserves internal punctuation and legitimate markup

3. **Brand term preservation**: Terms in `KEEP_VERBATIM` (e.g., "Active Inference", "InstituteOS", "Discord", "GitHub") are never translated.

### Incremental & Resumable

The translator is **incremental**: it only translates keys missing from the catalog (unless `--force`). It **checkpoints every 25 strings**, so a crash or interrupt loses almost nothing. Safe to run in the background.

```bash
npm run i18n:translate -- --all    # translates only missing keys
npm run i18n:translate -- --all --force  # re-translate everything
```

## How Localized Pages Are Generated

### Routing Architecture: One Central Point

All URL routing flows through **`urlDirForSlug()`** in `src/url-taxonomy.mjs`:

```js
// Locale-agnostic base dir
baseDirForSlug("about") → "about"
baseDirForSlug("project-foo") → "projects/foo"

// Active-locale-aware output dir
urlDirForSlug("about")     // English (default) → "about"
urlDirForSlug("about")     // Spanish active → "es/about"
urlDirForSlug("about")     // German active → "de/about"
```

Helper functions:
- **`localePrefix(code)`**: Returns `"<code>/"` for non-default locales, `""` for English
- **`baseDirForSlug(slug)`**: Locale-agnostic clean directory (no prefix)
- **`localeDirForSlug(slug, code)`**: Output dir under a specific locale
- **`stripLocalePrefix(currentDir)`**: Strip locale prefix for code inferring page structure

Because the prefix is applied to **both** the current page dir and every link target, relative-link math stays correct within a locale automatically, and `relPrefix` (asset paths) still counts the extra segment so images/CSS reach the true site root.

### Build Loop

In **`src/build.mjs`**:

```js
let pageCount = 0;
for (const locale of LOCALES) {
  setActiveLocale(locale.code);
  // Render every routed page once per locale
  for (const [slug, render] of slugRenderers) {
    writeFile(outputPathForSlug(slug), render());
    pageCount += 1;
  }
}
setActiveLocale(DEFAULT_LOCALE);

// Root-level singletons (emitted once, in default locale)
writeFile("404.html", layout(…));
writeFile("robots.txt", …);
writeFile("sitemap.xml", …);
writeFile("version.json", …);
writeFile("feed.xml", buildRssFeed());
writeFile("feed.json", buildJsonFeed());
writeFile("manifest.webmanifest", buildManifest());
writeFile("assets/js/search-data.js", buildSearchIndex());
```

**Result**: Every routed page is rendered `locales.length` times (currently 12). Root-level singletons (sitemap, feeds, search index, robots.txt, 404, version) are emitted once in the default locale.

## Locale Output URL Structure

### Current URL Structure

English (default): `/<path>/`
- Example: `/about/`, `/projects/foo/`, `/directory/`

Non-default locales: `/<code>/<path>/`
- Spanish: `/es/about/`, `/es/projects/foo/`, `/es/directory/`
- German: `/de/about/`, `/de/projects/foo/`, `/de/directory/`
- Arabic: `/ar/about/`, `/ar/projects/foo/`, `/ar/directory/`
- … (same pattern for all 12 locales)

### Sitemap & SEO

The sitemap (`sitemap.xml`) includes one entry per routed page **per locale**:

```xml
<urlset>
  <url>
    <loc>https://activeinference.org/</loc>
    <priority>1.0</priority>
    <changefreq>weekly</changefreq>
  </url>
  <url>
    <loc>https://activeinference.org/es/</loc>
    …
  </url>
  <url>
    <loc>https://activeinference.org/about/</loc>
    …
  </url>
  <url>
    <loc>https://activeinference.org/es/about/</loc>
    …
  </url>
  …
</urlset>
```

### Per-Page Alternate Links

Each page's `<head>` includes `hreflang` alternates for all locales, plus `x-default`:

```html
<link rel="alternate" hreflang="en" href="https://…/about/">
<link rel="alternate" hreflang="es" href="https://…/es/about/">
<link rel="alternate" hreflang="fr" href="https://…/fr/about/">
…
<link rel="alternate" hreflang="x-default" href="https://…/about/">
```

### Language Switcher

On every page, a CSP-safe `<details>`-based language switcher in the header (pure HTML, no JavaScript) allows users to jump between locales:

```html
<details class="lang-switcher">
  <summary>🌐 EN</summary>
  <div class="lang-menu" role="menu">
    <a role="menuitem" hreflang="en" href="../../about/">English</a>
    <a role="menuitem" hreflang="es" href="../../es/about/">Español</a>
    <a role="menuitem" hreflang="fr" href="../../fr/about/">Français</a>
    …
  </a>
</details>
```

Links are relative (calculated by `crossLocaleHref()`), so they work regardless of nesting depth.

### Machine-Translation Notice

Pages in locales with `"machine": true` display a visible provenance notice:

```
🌐 This page was machine-translated from English. View the English original.
```

The notice appears only on routed pages (not 404, feeds, or metadata) and links back to the English version.

## How to Add or Update a Locale

### Step 1: Register the Locale

Edit **`src/i18n/locales.json`** and add an entry under `locales`:

```json
{
  "code": "nl",
  "name": "Dutch",
  "nativeName": "Nederlands",
  "dir": "ltr",
  "machine": true
}
```

The `code` must be a valid BCP-47 language code (2–3 characters, lowercase). It becomes the URL prefix and `<html lang>` value.

### Step 2: Extract Source Strings (if Needed)

If no visible text has changed since the last extract, skip this. Otherwise:

```bash
npm run i18n:extract
```

This runs the build with `I18N_EXTRACT=1`, collecting every `tr()` source string into `src/content/i18n/_strings.json`.

### Step 3: Translate

Use the offline translator to generate the locale catalog:

```bash
# Local Ollama (default)
npm run i18n:translate -- --locale nl

# Or hosted API
I18N_PROVIDER=openai I18N_API_KEY=sk-... npm run i18n:translate -- --locale nl
```

The translator reads `_strings.json` and writes `src/content/i18n/nl.json`. It is incremental, so you can re-run it to fill in any missing keys.

For higher quality on larger models:

```bash
ollama pull aya-expanse:8b
npm run i18n:translate -- --locale nl --model aya-expanse:8b
```

### Step 4: Build & Verify

```bash
npm run build
npm run check
```

The build renders every page in every locale, and `check` verifies links, site contract, design system, and security policies. A partially translated locale is fine—missing keys fall back to English.

### Step 5: Set `machine: false` (Optional)

Once the translation is human-reviewed, edit `src/i18n/locales.json` and change `"machine": true` to `"machine": false` to remove the machine-translation notice.

### Step 6: For Subsequent Updates

If content or UI text changes and you want to re-translate:

```bash
npm run i18n:extract        # if strings changed
npm run i18n:translate -- --locale nl --force   # re-translate all keys
npm run build
npm run check
```

Or to incrementally fill in new keys:

```bash
npm run i18n:translate -- --locale nl   # translates only new keys
npm run build
```

## Writing Translatable Strings (for Contributors)

When adding visible UI text in a renderer:

1. **Wrap the English source once**, at the point the final text exists:

```js
import { tr } from "../i18n/index.mjs";
export function mySection() {
  return `<h2>${escapeHtml(tr("Open Source Map"))}</h2>`;
}
```

2. **Do not wrap**:
   - DOM IDs, anchors, URLs, slugs, class names
   - Arguments to helper functions that already call `tr()` internally (e.g., `sectionHeading`, `cardGrid`, `breadcrumb`)

3. **For interpolated values**, keep a placeholder:

```js
tr("View all {n} public repositories").replace("{n}", count)
```

This ensures the entire sentence translates as a unit, not individual fragments.

4. **After adding strings**, always run:

```bash
npm run i18n:extract        # updates _strings.json
npm run i18n:translate -- --locale <code>   # translates new keys
```

## Right-to-Left (RTL) Support

A locale with `"dir": "rtl"` (currently only Arabic) renders `<html lang="ar" dir="rtl">`, and layout uses logical CSS properties so the interface mirrors correctly. Full visual RTL polish of every component is a follow-up; the direction and content are correct, layout mirroring is best-effort.

## Limitations & Notes

- **Machine-translated locales** are flagged on-page with a link to the English original. Set `"machine": false` per locale after human review.
- **Search** uses a single English index (`assets/js/search-data.js`). Localized search is a future enhancement.
- **Coverage** varies: highest on the home page, curated content pages, and global chrome (nav, footer, buttons, breadcrumbs). Some specialized pages (directory, knowledge, resources tables) still contain English strings not yet wrapped in `tr()`; they fall back gracefully and can be wrapped incrementally.
- **All gates pass** with the full multilingual output: `npm run check` (links, site contract, design system, security).
